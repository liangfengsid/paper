Reviewer: 1

Recommendation: Author Should Prepare A Minor Revision

Comments:
Shuffle operation causes significant network transfer traffic especially for iterative applications which consisting multiple shuffle operations. The authors found that the keys are dependent in these iterations and proposes Key Dependency Graph that can capture the key dependencies and smartly partition the relevant keys into the same node. The experimental results show that the network traffic is reduced significantly. Overall, this finding and the proposed idea are very interesting and the implementations are solid.

This paper can be further improved by solving the following problems:
1. Have you considered the very special case in your system? For example, a huge single-row matrix (billions of numbers) multiples a huge single-column matrix, then will all element multiplications be mapped to one single node and executed there by using your key partitioning policy?
2. Have you utilized the compression file format provided by Spark in your experiment? Can you specify it in your experimental setup part? Can you compare the performance for compressed setting and uncompressed setting?
3. Can you compare the code for a simple application (e.g., matrix multiplication) in original Spark and your optimized Spark?

Reviewer: 2

Recommendation: Reject

Comments:
The paper presents an important problem: shuffle operations are known to be a substantial bottleneck of large-scale data parallel processing systems. The authors propose to explicitly define dependencies between keys across multiple computation steps. This is a very interesting topic, that has been discussed in many specific problems, such as parallel data joins, or BSP-style graph computations. Exploring models that can capture the problem in a generic way is worthwhile.

However, the current manuscript is scientifically flawed; The contributions are vaguely defined, with several important aspects not clearly defined. The paper does not demonstrate what is the applicability of the proposed approach, moreover when the initial step requires manual code inspection and definition of the graph. Additionally, the actual benefits of adopting this model are not shown in a convincing manner.  

Map Reduce is not the most appropriate abstraction for motivating the KDG. First, MapReduce is restricted to a single Map/Shuffle/Reduce cycle, and as the authors discuss, the framework itself is not suitable for applying this approach. Additionally, while Map Reduce is widely understood, the Reduce part of it is actually a compound operation. Reduce in MapReduce comes immediately after a Shuffle/GroupByKey, followed by a Reduce of the grouped values. The key part for the authors proposal is this shuffle part, but that is not explicitly named in the description provided in Section 2.

The CDG contribution should be expressed in more generic and accurate terms. Additionally, the claim that this technique works in approach is exclusively valid for iterative computations. Any dataflow where there are multiple stages of shuffling, or GroupByKey, can concievable benefit from this approach. Additionally, the paper should discuss whether the nature of other types of operations (Map, Reduce) actually matter or not for what it is being proposed. Intuitively, it seems that any Map-like operation (in the case of Spark Map, FlatMap, Filter), as well as the Reduce will be embarrasingly parallel. However, there are some cases, such as some functions applied to FlatMap, where the key space will change, hence having A'i =/= Ai-1. The formulation of the abstraction should be thoroughly revisited, in order to clarify these concerns. There should be clear conclusions of the applicability of the KDG, as well as its limitations. It is impossible to know currently whether joins can be supported, or whether the number of partitions is relevant in any way to the presented approach.

The definition of a Confluence Graph is not expressed clearly enough. Authors claim that the preferred way to state dependencies is through patterns, but these patterns are not formally defined in any part of the paper. Patterns are claimed to be critical, as otherwise the complexity of building and implementing the graph would make this approach scale poorly. The usefulness of the algorithm described in Fig 2 is unclear. The algorithm directly maps key dependencies into a graph. However, it does not provide any insight in how to resolve patterns.  

The graph needs to be manually defined, which greatly limits the applicability of the approach. Moreover, explicit mapping to computing frameworks also needs to be specified as part of the Confluence. Key mapping depends on finding pure confluence subgraphs, which might not appear in many problems.

It is not clear how the mapping works, in particular because sections 3.4 and 3.5 are very difficult to read as written. All the discussion covers a single pure confluence graph, but the section should describe the whole process of mapping a complete DKG. Otherwise, it is impossible to know the expected benefit. 3.5 mentions later stages are unpredicable, which adds to the confusion about the impact of this technique.

The choice of applications/algorithms does not seem representative enough of complex parallel data algorithms. Some choices are misleading, such as MovieLens ALS (The code that is being analysed is a sample of trained models with differing parameters, which has no real dependencies between the sets computed at each iteration, other than for computing a maximum). Adjacency list is a very unusual graph algorithm, whereas there is no mention of the main parallel graph computations using BSP-style graph traversals, such as PageRank, or Shortest Path.

Validation could be improved in order to obtain further insight. Shuffle size figures could be compared to job execution time figures, in order to estimate the actual performance impact of this type of optimisation. There is no mention of the number of partitions used in each experiment, moreover with relatively small datasets for a 16 node cluster. Experiments exploring variation in the number of partitions would also provide additional evaluation of the approach.

The paper is reasonably written. However, there are several aspects in which grammar errors complicate understanding.

Additional detailed comments follow:

The example in the abstract takes an excessive amount of space and it is a well known issue.

pg 1 In the introduction, YARN and Mesos are mentioned as computing frameworks, whereas they are only resource managers.

pg 1 There should be a reference on the impact of shuffle traffic in network operations. Additionally, this should be contrasted with the Ousterhout et al NSDI 15 paper that makes the opposite claim for Spark.
Data are stored in disks in "some" traditional distributed paradigms. Which are these paradigms? how prevalent are they? "some" is too vague for a scientific statement.

Page 2: clarify "KDG does not have to be accurate" I presume this implies that at worst, performance will be degraded to similar levels to a random partition,but this is not mentioned at this point of the manuscript.

Contribution 1: KDG is not a data structure.

In many instances, the authors use the article 'the' inappropriately.

2.3 should make explicit the restriction that Ai and A'i are the same set of keys in the equations.

Figs and could be represented differently. A separate plot purely for standard deviation is not that meaningful. It would be possible to explore alternative representations (e.g. adding error bars to the first plot, while destacking the bars)

Section 5.2 can be greatly shortened. The discussion does not mention systems such as Hadoop, or the work by Jimmy Lin on enabling iterative computations with Hadoop.

Section 5.4 could be reduced to a single paragraph, as it mostly provides implementation details.

Reviewer: 3

Recommendation: Author Should Prepare A Major Revision For A Second Review

Comments:
This paper presents Confluence, an improvement over Spark for reducing unnecessary shuffles by leveraging the dependencies in partitioning keys.

Here is the strong point of the paper:

S1. The paper designs and implements a programming interface extension for application programmers to expose the partitioning properties of their distributed Spark programs. Adding such an API and corresponding optimizations into distributed data processing frameworks like Spark and Hadoop would be useful.

However, this paper has the following issues:

W1. The technique for inferring dependencies of partitioning properties and reducing unnecessary repartitioning steps (a.k.a., shuffles) is not new. The SCOPE optimizer covers most state-of-the-art techniques for that (“Incorporating Partitioning and Parallel Plans into the SCOPE Optimizer”, published in ICDE’10). The SCOPE optimizer incorporates partitioning properties as part of the data properties that a query optimizer understands and uses a Valcano-style (“Volcano - An Extensible and Parallel Query Evaluation System”, published in TKDE’94) top-down strategy to enforce the partitioning properties. To some extent, the core technique presented in this paper falls into a subset of the SCOPE paper, as the latter covers a spectrum of data redistribution patterns. Regardless of that, this paper still have some contribution, i.e., how to build a useful API for programmers to express partitioning properties of their programs to the Spark/Hadoop framework.

W2. The theoretical analysis for shuffle data size (e.g., formula (5), Theorem 2, and Table 2) is obvious and is not very useful. Particularly, n usually is not a small number and hence (n-1)/n can almost be thought as 1. I suggest authors remove those analysis but focus more on the API.

W3. The experimental evaluation uses very small datasets, e.g., each individual dataset (20~60 million data points) can fit into a single machine used in for the experiments (32 GB DDR RAM). The shuffle sizes shown in Figure 5, 6 and 7 also show that the data involved in the experiments are tiny. Dataset size might not be a big issue for shuffle size experiments, given that the reduced shuffle size can be roughly estimated without actually running the algorithms. But it would be very informative if authors can report the end-to-end response time improvement given by the shuffle optimization -- in that case larger datasets will make more sense.

W4. In Section 3.6, authors claim that wrong KDGs will not affect the program correctness.  This looks plausible. For example, if a shuffle step is removed by mistake, how can the program still deliver correct results?  Please explain more details in the next revision.

W5. In Section 4.3, authors discuss the K-means algorithm. However, a typical K-Means implementation on Hadoop and Spark broadcasts cluster centers in each iteration, and hence the input data do not need to be shuffled. In that regard, I don’t think K-means is a good motivating example for the shuffle optimization.

W6. In Section 5.2, authors claim that Confluence cannot be applied to Hadoop. This is not entirely true, as contiguous Hadoop MR jobs can be pushed into the first MR job if the partitioning keys are compatible, just that the first reduce implementation need to call a pipeline of consequent map/reduce functions as well as perform the grouping of reduce keys.
